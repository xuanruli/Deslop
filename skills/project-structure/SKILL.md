# 代码组织原则

## 核心心法

**目录结构反映领域模型，而不是文件类型。**

错误的分法（按文件类型，反复出现的反模式）：
```
src/
├── utils/        # 什么都往里塞
├── helpers/      # 同上
├── components/   # 所有组件平铺
├── services/     # 所有服务平铺
└── types/        # 所有类型平铺
```

正确的分法（按概念/领域分层）：
```
src/
├── <底层原语>/        # 通用、无业务的基础设施（异步原语、数据结构）
├── <框架/平台>/       # 在原语之上的执行/调度框架
├── <领域模块 1>/      # 每个领域是一个独立子包
├── <领域模块 2>/
├── <领域模块 N>/
└── <顶层胶水文件>     # 把领域组合成应用
```

读者扫一眼目录就该能猜到这个项目是干啥的。如果看到的是 `utils/services/components`，说明设计者还没想清楚。

## 原则 1：目录名是名词，一个目录 = 一个概念

好名字：`persistence/`、`realtime/`、`scheduler/`、`auth/`、`payments/`

差名字：`utils/`、`helpers/`、`common/`、`shared/`、`misc/`、`lib/`（作为顶层时）

**判断标准**：能不能用一句话回答"这个目录里的代码负责什么单一职责"？答不出来就是垃圾桶目录。

**例外**：在某个领域目录**内部**有一个小 `utils.py`/`utils.ts` 放该领域专属的辅助函数是可以的（作用域受限）。但**顶层 `utils/`** 几乎总是设计失败的信号。

## 原则 2：一个文件 = 一个概念

每个文件名是一个名词，导出一个核心 abstraction。

好：`channel.py`（一个 channel 类型）、`rwlock.py`（一个读写锁）、`multi_queue.py`（一个多队列）

差：`stuff.py`（5 个不相关的函数）、`misc_helpers.ts`（杂物）、`index.ts` 里塞 800 行业务逻辑

**判断标准**：能不能给这个文件起一个明确的、单数的名字？多个互不相关的东西塞一个文件 → 拆开。

**例外**：高度耦合的几个小类型（一个 enum + 它的几个工厂函数）可以同文件，因为分开反而难读。判断"耦合度"，不是"代码行数"。

## 原则 3：按抽象层级自底向上分层

健康的代码库有清晰的**依赖方向**：

```
顶层胶水（应用入口、agent.py、main.ts）
    ↓ 依赖
领域模块（auth、payments、search、chat）
    ↓ 依赖
框架层（scheduler、graph、event bus）
    ↓ 依赖
原语层（async primitives、collections、I/O）
```

**判断标准**：
- 底层不能 import 上层
- 同层之间最好也不互相 import（除非真有关系）
- import 关系画出来应该是 DAG（无环），不是网状

如果发现"循环依赖"或"底层 import 了业务模块"，说明分层错了。

## 原则 4：每个子包有"公开 API 面"

子包根目录的 `__init__.py`（Python）或 `index.ts`（TS）**只 re-export 公开的东西**。内部实现文件存在，但不出现在公开 API 里。

```
my_package/
├── __init__.py        ← 只 re-export Chan, ChanClosed, RWLock 等公开类型
├── channel.py         ← 实现细节，外部不直接 import
├── rwlock.py
└── _internal.py       ← 下划线开头 = 明确内部
```

消费者只 `from my_package import Chan`，不 `from my_package.channel import _ChanState`。

**判断标准**：能不能从 `__init__.py` / `index.ts` 一眼看出"这个子包对外提供什么"？

**为什么重要**：内部实现可以重构而不破坏调用方。如果调用方满世界 import 内部文件，重构就动不了。

## 原则 5：类型/Schema 单独成文件

每个有自己类型体系的子包都有 `types.py` / `types.ts`。

```
auth/
├── __init__.py
├── client.py       # 实现
├── handlers.py     # 实现
└── types.py        # 类型定义（dataclass、TypedDict、enum、Protocol）
```

**为什么**：
- 类型变更频率低于实现 → 隔离开避免不必要的 import 牵连
- 跨模块共享时，import `types.py` 不会拖入业务逻辑
- 文档/审阅时只看 `types.py` 就能掌握"这个模块的数据形状"

## 原则 6：跨边界的契约必须显式声明

任何两个组件之间的边界（进程间、语言间、服务间、模块间），它们之间的**消息格式 / 数据契约**必须有一份显式定义。

实现方式（按强度排序）：
- Schema 库（Zod / Pydantic / Protobuf）→ 运行时 + 编译时校验
- 类型定义文件（TS interfaces、Python TypedDict）→ 编译时校验
- 文档约定（README、注释）→ 只有人脑校验，最脆弱

**判断标准**：边界两边的代码改了一边，能不能在 CI 抓到不一致？抓不到就是没契约。

## 原则 7：入口文件是"组合器"，不是"业务实现"

应用的 `main.ts` / `index.ts` / `app.py` 应该读起来像"招标书"：声明依赖、组装、启动。**业务逻辑不在这里**。

好的入口：
```ts
init(monitoring);
const modules = registerModules([Auth, Storage, Scheduler]);
const plugins = registerPlugins([passthrough, screenshot], modules);
await start(modules, plugins);
```

坏的入口：1000 行 if/else + 业务代码 + 各种 inline handler。

**判断标准**：入口文件应该能在 30 秒内读完，并理解"这个应用大致由什么组成"。

## 原则 8：跨函数共享的上下文用 Context，不要参数透传

如果同一个值要传过 5 层函数才能到使用方，这是设计 smell。改用：
- Python：`contextvars.ContextVar`
- Node/Bun：`AsyncLocalStorage`
- React：Context API
- Go：`context.Context`（语言内置）

**判断标准**：超过 3 层透传的参数（例如 user/tenant/request-id），考虑放进 context。

**反面**：不要把所有东西都塞 context，那会变成隐式全局状态。**临时需要 = 参数；贯穿请求生命周期 = context**。

## 原则 9：单一事实源（SSOT）

任何"两处必须保持一致"的东西，必须从一处派生。常见场景：

| 场景 | SSOT 实现 |
|---|---|
| 数据库表 + 前端类型 | Schema 文件 → 代码生成 |
| 主进程注册 + 渲染端事件类型 | 一个 registry 文件 |
| 路由 + 类型化 client | 一个路由表 |
| 环境变量 + 校验 + 类型 | 一个 env schema 文件 |
| 错误码 + 错误信息 + i18n key | 一个错误定义文件 |

**判断标准**：找到任何"改 A 必须记得改 B"的地方，那就是 SSOT 设计漏了。

## 原则 10：不要预先抽象

新功能的第一次实现**留在使用它的地方**。出现第二个使用者时再抽出来。

```
错误：刚开始就把所有 "可能复用" 的代码抽进 shared/
正确：先放在 features/foo/，等 features/bar/ 也要用时再抽
```

**为什么**：预先抽象往往抽错维度。第二个 use case 出现时，你才知道"真正需要被参数化的是什么"。提前抽 → 接口设计错 → 第二个 use case 来时要么硬塞要么重写。

**反面**：明显的通用工具（日期格式化、字符串处理）该抽就抽，不要僵化教条。**判断核心**：这段代码包含业务知识吗？包含 → 等第二个 caller；纯技术 → 可以提前抽。

## 原则 11：测试代码靠近被测代码

```
foo/
├── __init__.py
├── channel.py
├── channel.test.ts (或 test_channel.py)
└── rwlock.py
```

或：
```
foo/
├── src/
└── tests/   # 镜像 src 的结构
```

**反模式**：所有项目所有测试都堆在仓库根 `tests/` 下。找测试要全仓搜，改实现忘改测试。

**判断标准**：改一个文件，对应的测试应该一眼能看到。

## 原则 12：配置抽包，不要复制

如果两个子项目共享 lint 规则、tsconfig、CI 步骤——把它们抽成可被 import 的配置包/模板，而不是复制粘贴。

复制配置 = 3 个月后必然漂移 = 升级时改 N 处。

**判断标准**：同一份配置出现在 2+ 地方？→ 抽。

## 原则 13：包管理器的选择有差异，不要默认

不同生态的包管理器**不是等价的**，选错会影响整个项目的体验和正确性：

| 生态 | 推荐 | 关键优势 |
|---|---|---|
| JavaScript/TypeScript | **pnpm**（不是 npm/yarn） | 严格隔离 + 内容寻址硬链接 + 原生 workspace + 防幽灵依赖 |
| Python | **uv**（不是 pip/poetry） | 速度数量级提升 + workspace + 锁文件 |
| Rust | cargo | 内置，原生 workspace |
| Go | go modules + workspace | 内置 |

**判断标准**：新项目不要假设"哪个都行"。每个生态选当下**最严格、最快、最有 workspace 支持**的那个。

## 原则 15：用 workspace 管理本仓内部包

任何成熟的包管理器都有 workspace 概念，它解决三件事：依赖去重、内部包互引（本地软链接，不走外部 registry）、一条命令打理全仓。

**核心约束**：本仓内部包之间的依赖**用 workspace 协议**引用，不写发布版本号。否则会失去"改一处全仓立即生效"的核心优势——这是 monorepo 之所以是 monorepo 的原因。

## 原则 16：顶层目录按"角色"二分

最稳定的 monorepo 顶层就两个桶：

- **可被独立部署/交付的东西**（产品形态）
- **被前者 import 的东西**（可复用单元）

后者再按职责细分（参考原则 1、3）——分类标准是它的**领域职责**，不是它的形式。

**依赖只能单向流动**：被部署的 → 被 import 的 → 更底层的被 import 的。反向 = 循环依赖 = 失败。

判断标准：能独立交付？→ 部署桶。被别人 import？→ 复用桶。

## 原则 18：发布/部署/构建脚本必须可分阶段重试

任何"耗时长 + 可能失败 + 失败不影响前面"的流程，拆成可单独跑的 phase：

```
release.sh build       # 5min，本地能跑
release.sh sign        # 1min，需要密钥
release.sh notarize    # 30min，云端，可能超时
release.sh publish     # 5min，需要网络
```

而不是 `release.sh all-in-one`。

**判断标准**：某一步失败，能不能从该步重试，而不是从头来？

## 原则 19：默认私有，显式公开

每个模块/包应该有"公开 API 面" vs "内部实现"的明确区分。利用语言自身的可见性机制（命名约定、显式 export 列表、包的 exports 字段等）让内部的东西**看起来就是内部**。

**反面**：默认全部公开 → 调用方满世界 import 内部细节 → 重构时动不了任何东西。

**判断标准**：你能把某个文件改名 / 拆分 / 内联，而不影响外部调用方吗？能 → 内部边界设对了。

## 原则 20：组织知识沉淀进 repo，分层放

可搜索、可被 AI 读取的 markdown 文件比聊天/wiki 有复利。一个成熟项目通常有这几类文档，按受众和粒度分开：

- **入门文档** → 怎么 setup、怎么跑、面向新人
- **架构文档** → 设计决策、模块关系、面向想理解全局的人
- **运维手册（runbooks）** → 操作步骤，面向值班的人
- **复盘（postmortems）** → 生产事故记录，面向"避免重蹈覆辙"
- **局部上下文** → 每个子包自己的简介，面向只在这个子包工作的人/AI

**为什么**：聊天工具里的讨论一周后就找不到，markdown 进 repo 才能 grep、链接、被 AI 检索。

## 决策模板

新建一个东西时按这个顺序问自己：

1. **这是什么领域 / 抽象层级？** → 决定放哪个父目录
2. **这是一个单一概念吗？** → 决定独立文件还是合并
3. **它对外的 API 是什么？** → 决定公开什么、私有什么
4. **它依赖什么？** → 检查依赖方向是否违反分层
5. **它和谁共享数据契约？** → 决定 schema/types 放哪
6. **有没有第二个使用者？** → 决定要不要抽象，还是先内联

## 反模式快速识别

看到这些信号，结构设计大概率出问题：

- 顶层有"无领域含义"的容器目录（utils、helpers、common、misc、shared）
- 单个文件远超合理长度，且内容是多个不相关概念
- 一个目录平铺过多文件却没有任何分组
- 入口文件里塞业务逻辑而不是组合调用
- 跨层级 import（业务 import 入口、底层 import 上层）
- 同一份配置 / 常量 / 类型出现在多个位置（SSOT 失败）
- 测试要全仓搜才能找到对应代码
- 函数参数过多，且其中一部分本该是 context 承担
- 改一个小字段要动多处文件（SSOT 没设计好）
- 包之间引用形成环（依赖图非 DAG）
